引入

现在大多数企业都采用微服务模式，在生产环境中，经常会遇到网络抖动。一些RPC框架中在框架级别实现了重试机制，例如dubbo，但是这依赖于公司中间件部门比较强大，可以灵活配置。如果遇到需要重试的问题，中间件不能满足自己需要，需要自己实现的时候，需要考虑什么问题呢？

场景

重试的目的一般是保证可用性。无论是读写，重试都要保持幂等，这是第一原则。

  • 读接口是幂等的，所以适合使用重试策略。
  • 简单写接口。就要格外注意，比如笔者最近遇到的定时任务订单结算打款业务场景，如果打款接口调用失败，会进行重试，这时候就要求打款业务保持幂等，这个场景下较为简单，生成一个业务打款ID就可以。
  • 复杂写接口。比如一个方法，你耦合的写逻辑较多，需要依赖多个外部服务，这个时候是不适合重试的，可能需要你从业务上拆分下。

一旦你确定了要用重试，就需要考虑以下问题：

  1. 什么条件下重试
  2. 什么条件下停止
  3. 如何停止重试
  4. 停止重试等待多久
  5. 如何等待
  6. 请求时间限制
  7. 如何结束
  8. 如何监听整个重试过程

本文就这几个问题，介绍几个框架及其实现原理，并比较其特性，以便读者选择。
